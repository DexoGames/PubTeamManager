# New Systems ŌĆö Build & Wiring Guide

This doc covers everything you need to do **in the Unity editor** to light up the systems added in this
pass. All the C# is written and compiles; the items below are scene objects, prefabs, button hookups and
(optional) authored assets. Each section says what's **required** vs **optional**, and what already works
without any editor work.

The features:
1. Tactics overhaul ŌĆö mentality (defensiveŌåöattacking), reliances, familiarity, complexity/intelligence
2. In-match tactical changes (anytime mentality nudges + half-time structural window)
3. Injuries, suspensions & death
4. Interview day (gating, squad cap, personality clues)
5. Team-talk minigames (half-time)

> **Design rule baked into all of it:** every edge carries a matching risk. Player dependencies amplify a
> player's weaknesses as well as his strengths; complex tactics need intelligent players; changing your
> setup burns Familiarity; a botched team talk *lowers* morale. Tune the numbers to taste (see "Tuning").

---

## 0. What already works with NO editor changes

- **Tactics maths**: mentality, dependencies, the muscle-memory familiarity model, complexity and the
  squad-average intelligence gate all run inside the match engine already. Even before you build any new UI, a
  loaded/new game uses them (defaults: Balanced mentality, no dependencies, Familiarity ~50% climbing toward 100%
  as you play/train an unchanged tactic).
- **Tactic persistence**: formation, instructions, dependencies, mentality and familiarity now save/load
  (added to `TeamConverter`). No editor work.
- **Formations shape the tactic** (no wiring): each formation's authored `FormStats` now base-replaces the
  tactic's Complexity/Intensity/Control/Threat/Security, and its `Tempo` drives build-up directness in the match
  engine. Tune by editing the `FormStats` on each formation asset (`Resources/Formations/Usable/*.asset`).
- **Player availability** (injury/suspension/death fields) saves/loads automatically (inline `Player`).
- **In-match fouls/cards/injuries** are generated by the match engine and shown in the live feed via the
  existing `MatchEventUI` (they're just `Highlight`s) ŌĆö no new prefab needed for the feed text.

The things below make these **visible and controllable** to the player.

## 1. InjuryManager (REQUIRED for injuries/suspensions)

Injuries, suspensions and death are driven by `InjuryManager`, a singleton that must exist in the scene.

1. Select your **managers GameObject** (the one holding GameManager, EventsManager, etc.).
2. **Add Component ŌåÆ InjuryManager.**
3. Inspector fields:
   - **Daily Misfortune Chance** (default `0.02`) ŌĆö chance per day a squad member gets an off-pitch
     injury/illness. Lower = fewer injuries.
   - **Injury Event / Illness Event / Death Event** (`EventType`) ŌĆö *optional*. If assigned, injury news
     is routed through the existing morale **inbox** (and a notification) and applies the EventType's
     `moodChange`. Leave null to just apply a morale shock + log + add to `RecentNews`.
   - **Notification Prefab** ŌĆö *optional*; the same notification prefab EventsManager uses. Only used if an
     EventType above is also set.

### Optional: author the 3 injury EventTypes (recommended)
Create via **Assets ŌåÆ Create ŌåÆ EventType** under `Resources/Events/` (so they round-trip and can be
discussed). Suggested values:

| Asset | tag | description | severity | moodChange | noAffected |
|---|---|---|---|---|---|
| `InjuryEvent` | Special | `<1> has picked up an injury and faces a spell out.` | Unfortunate | -4 | 1 |
| `IllnessEvent` | Special | `<1> has come down ill and will miss some time.` | Unfortunate | -3 | 1 |
| `DeathEvent` | Special | `The club is in mourning after the tragic loss of <1>.` | Dire | -25 | 1 |

`<1>` is replaced with the affected player's name (existing token system). Assign these to the matching
InjuryManager slots.

### Morale impact scales with severity (automatic)
`InjuryManager.ApplyInjuryMorale` sizes the morale hit by injury type: a **Knock** only nicks the injured
player's own mood (small, no squad effect); **Standard/Hamstring** hit him more with a small ripple to teammates;
an **ACL** (months out) hits him hard and visibly dents the squad; a **death** rocks the whole team. If you assign
the optional EventTypes above, keep their `moodChange` small (e.g. 0 to ŌłÆ3) since this direct, severity-scaled
shock is the main driver ŌĆö otherwise the victim is double-counted. Tune the bands in `ApplyInjuryMorale`.

### Show availability in the squad UI (recommended)
`Player` now exposes:
- `IsAvailable`, `IsInjured`, `IsSuspended`, `IsDeceased`
- `AvailabilityStatus` ŌåÆ e.g. `"Injured (Hamstring, back ~12 Jun)"`, `"Suspended (2)"`, `"Deceased"`.

Add a TMP text (or icon) to your **PlayerRowUI / SquadStatusWidget** that reads `player.AvailabilityStatus`
and tints red when `!player.IsAvailable`. The `SquadStatusWidget` already flags fatigue/low-morale ŌĆö extend
its `Setup()` to also list unavailable players (use `team.AvailablePlayers` / `team.UnavailableStarters`).

> Lineups auto-fix: `Team.EnsureAvailableLineup()` is called before every match and benches anyone who
> can't play, swapping in the best-rated available sub. Deceased players stay in the squad list (marked
> Deceased) until you release them ŌĆö you may want a "Release" affordance for that.

## 2. Tactics page additions

The tactics page (`TacticsPageUI`) already has the formation editor and instruction toggles
(`TacticGridLayout`). Add the following.

### 2a. Mentality selector (defensiveŌåöattacking dial) ŌĆö `MentalitySelectorUI`
1. Create a **Slider** on the tactics page.
2. **Add Component ŌåÆ MentalitySelectorUI**; it auto-configures the slider to whole numbers 0ŌĆō6.
3. Assign a TMP **label** field (shows "Ultra-Defensive ŌĆ” Balanced ŌĆ” All-Out Attack").
4. Done ŌĆö moving it sets `Tactic.BaseMentality` and refreshes the stat sliders.

### 2b. Familiarity slider (non-interactable) ŌĆö `FamiliaritySlider`
1. Create another **Slider** (styled like the other tactic stat sliders).
2. **Add Component ŌåÆ FamiliaritySlider** ŌĆö it forces the slider non-interactable, sets range 0ŌĆō100, and shows
   the team's current Familiarity. Optional label child auto-set to "Familiarity"; the fill colour is configurable.
3. It refreshes on show and on any tactic edit. (Familiarity itself only climbs after matches/training ŌĆö see ┬¦0/┬¦6.)

### 2c. Tactic info / trade-off read-out ŌĆö `TacticInfoUI`
1. Add a TMP text somewhere on the page.
2. **Add Component ŌåÆ TacticInfoUI**, assign the text.
3. It shows Familiarity %, current mentality, Complexity, and the **squad-average intelligence vs the bar** ŌĆö i.e.
   whether the XI is collectively smart enough (a smart side covers for one dim player; only if the *average* falls
   short do below-bar players get penalised). This is how the player *sees* the complexity trade-off.

### 2d. Reliances ŌĆö now a property of instructions
A **reliance** is authored **on a `TacticInstruction` asset** (it *becomes* a reliance by filling in its **eligible
position groups** ŌĆö no separate flag ŌĆö then set its amplified `PlayerStat`s + multiplier and a **familiarity
penalty**), so it sits in
`TacticGridLayout` like any other instruction. Toggling it on opens the shared picker (filtered to eligible
starters) to choose the **one** reliant player; the binding is by **squad slot**, so it follows subs and survives
formation changes, and auto-disables if the player moves into a disallowed group. **Full steps in ┬¦8.2** (plus the
shared `PlayerPickerPopup`, ┬¦8.1).

> Author the trade-offs: `eligibleGroups` keeps a reliance sensible (e.g. crosses ŌåÆ wide/forward only) **and now
> also scopes *when* it bites** ŌĆö the match engine plays out position-band phases (build-up Ōēł defenders, finishing Ōēł
> attackers, transitions Ōēł the unions), and a reliance only sways phases whose band sits inside its groups (a
> defenders reliance lifts the build-up, not the finish). `familiarityPenalty` is how much *changing* the reliant
> player (a sub) dents familiarity (0 for minor ones); and normal `statModifications` (e.g. +Complexity) make
> leaning on a player also raise the squad-IQ demand.

### 2e. (Optional) more named instructions for variety
The mentality dial already gives the full defensiveŌåÆattacking range. If you also want extra **named**
instructions (HighLine, LowBlock, CounterPress, etc.), author them as before:
**Assets ŌåÆ Create ŌåÆ Tactic ŌåÆ Instruction** in `Resources/Tactics/Instructions/`, set their `statModifications`
/ `tacticDependencies` / `incompatibleInstructions`, then add them to the `TacticGridLayout.tactics` list.
They flow through the new system unchanged (and now persist).

**Synergies (`complementaryInstructions`).** To reward pairs that work well together, fill an instruction's
**Complementary Instructions** list: each entry is a **partner instruction** + a list of **tactic-stat perks**
(a `TacticStat` + amount, just like `statModifications`). When *both* the owner and the partner are active, the
owner's perks are added (applied as step "1b" of `RecalculateStats`, after base instruction stats, before
mentality). It's **one-directional** ŌĆö e.g. give *High Press* a perk that only lands when *High Line* is also on;
list the synergy on both assets if you want each to perk the other. (This is the positive mirror of
`incompatibleInstructions`.) Keep the design rule in mind: a perk is an upside, so consider an authored downside on
the same instruction ŌĆö e.g. High Press + High Line gains Threat but the High Line already carries a Security cost.

### 2f. Complexity slider colour + squad-IQ warning ŌĆö `ComplexitySlider`
The squad-IQ feedback lives in its **own component** so plain stat sliders carry none of it. On the **Complexity
slider only**, swap its `TacticStatSlider` for **`ComplexitySlider`** (Remove Component on the old one ŌåÆ Add
Component ŌåÆ ComplexitySlider). It's a subclass, so it still shows the value like any stat slider, and it forces its
own `stat` to Complexity (the inherited `stat`/`color` fields are unused ŌĆö ignore them).
1. Optionally set **Ok Color** (green) and **Too High Color** (red) ŌĆö sensible defaults are baked in, so you can skip this.
2. Make a text label (e.g. *"ŌÜĀ Squad IQ too low for this tactic"*) and place it as a **sibling just above the
   slider ŌĆö NOT a child of the slider** (the base grabs the first child TMP as the stat-name label, so a child label
   would get overwritten). Assign it to the **Squad Iq Too Low Warning** field (optional ŌĆö the colour works without it).
3. That's all. The fill turns red and the label appears whenever the on-pitch XI's **average** intelligence is below
   the complexity bar (`CurrentStartingIntelligence() < IntelligenceThreshold`, matching the in-match penalty);
   green and hidden otherwise.

It refreshes live as you change instructions, mentality, formation, **or swap players** ŌĆö formation changes and
player swaps now fire `OnTacticChange`, so every stat slider + `TacticInfoUI` updates after them too.

## 3. In-match tactical changes (`MatchSimPageUI`)

`MatchSimPageUI` gained these hooks (all optional to wire, but they're the request-2 feature):

**Rule:** during open play you can ONLY shout / change mentality; full structural changes (formation,
instructions, dependencies) are **half-time only**. Any shout/mentality change takes effect at the **start of the
next simulated minute** (the engine reads the tactic fresh each minute ŌĆö already the case, no wiring needed).

### 3a. Anytime "More Attacking / More Defensive" + mentality slider
1. Add two **Buttons** to the match screen (visible during play).
2. Hook their OnClick to `MatchSimPageUI.MoreAttacking()` and `MatchSimPageUI.MoreDefensive()`.
3. (Optional) Add an in-match **mentality Slider** (whole numbers 0ŌĆō6) whose OnValueChanged calls
   `MatchSimPageUI.SetMentalityInMatch((int)value)` to jump straight to a level.
4. Add a TMP text and assign it to the **Mentality Text** field on MatchSimPageUI ŌĆö shows the live mentality and
   updates on each change. A touchline-shout line is also printed into the match feed.

These are temporary: the in-match shift resets to your base mentality at full time.

### 3b. Half-time panel (full tactics + team talk) ŌĆö now wired end-to-end
1. Create a panel GameObject (e.g. `HalfTimePanel`) on the match screen; assign it to the **Half Time Panel** field
   on MatchSimPageUI. It auto-shows only at half-time.
2. Inside it put: a **Team Talk** button (section 5) and an **Edit Tactics** button whose OnClick calls
   `MatchSimPageUI.OpenHalfTimeTactics()`.
3. On the **tactics page**, add a "Resume Match" button: assign it to the new **Resume Match Button** field on
   `TacticsPageUI`, and hook its OnClick to `TacticsPageUI.ResumeMatch()`. It auto-shows only when the page was
   opened from half-time.

Flow (no extra code needed): Edit Tactics ŌåÆ full tactics page (structural edits allowed because `IsHalfTime` is set)
ŌåÆ Resume Match ŌåÆ `UIManager.ResumeMatchFromTactics()` ŌåÆ match screen re-appears in the half-time state with your
edited tactic, and "Begin Second Half" continues. The match is never restarted.

## 4. Half-time team talk ŌåÆ see **`TEAM_TALK_UI.md`**

The team talk is now a **choose-a-tone** screen that **reuses the 1-on-1 discussion reaction system**, broadcast to
the whole squad: you pick one `Event.Response` (Praise/Rage/Encourage/ŌĆ”), each player reacts via the shared
`EventsManager.ReactionTable` + `Person.NewMorale`, and the **severity comes from the half-time scoreline**. Every
squad member shows as a morale box along a shallow arch with per-player reaction feedback. `TeamTalkController` lives
on the managers object; full editor wiring (prefabs, arch params, hooking the Team Talk button ŌåÆ `TeamTalkUI.Open`)
is in **`TEAM_TALK_UI.md`**.

> The older microgame path (`TeamTalkMinigame`/`GerrymanderGame`/`BalanceGame`, `TeamTalkController.CreateRandom`/
> `ApplyResult`) is left in the codebase but **dormant** ŌĆö not wired into the active UI.

## 5. Interview day (`RecruitmentPageUI`, `InterviewManager`)

Most of this works already; finish the question UI and the release-before-hire flow.

> Full Inspector checklist (every serialized field + OnClick) is in **`WIRING_RECRUITMENT_HALFTIME_SIM.md`**;
> the answer-bucketing design is in **`INTERVIEW_INMATCH_TEAMTALK_BUILD.md`**.

### 5a. Gating (already wired)
- Interviews are only available on a scheduled interview day (`RecruitmentManager.CanInterviewToday`),
  one session per day. `RecruitmentPageUI` shows a "No Interviews Today" message otherwise.
- The schedule's `OnInterviewDay` is subscribed (new game + load) ŌåÆ tops up the candidate pool.
- **Testing:** tick `RecruitmentManager.debugEndlessInterviews` to make every day an interview day with unlimited
  sessions (off for normal play).

### 5b. Question buttons ŌĆö `InterviewQuestionButton`
1. Build a row/grid of **Buttons** in the interview UI.
2. On each, **Add Component ŌåÆ InterviewQuestionButton**, pick a **Question Type** (and a **Stat** if it's
   `AskAboutStat`), assign a TMP **label** (auto-fills with the question text), and hook OnClick ŌåÆ `Ask()`.
3. Suggested set:
   - Ability: `BiggestStrength`, `BiggestWeakness`, a few `AskAboutStat` (Shooting, Pace, Passing, Strength, HeightŌĆ”),
     and **`CompareToPlayer`** (opens the player picker to compare against any squad member ŌĆö no Stat needed).
   - **Personality probes**: `HandleCriticism`, `WorkEthic`, `BigGameMentality`, `Leadership`.

Every question now keys its answer on a ~3-way **response bucket** (different bucketing per question), and **every**
answer narrows the hidden personality (intersection), shown in the questions-remaining text
(`InterviewManager.NarrowedPersonalitiesText()`), e.g. *"Personality: Cocky / Aggressive"*. Ability answers also vary
by **stat shape** (clear standout vs all-rounder), so they reveal ability *and* a personality clue. Answers appear in
the existing **DialogueUI** (wired by `InterviewManager.StartInterview`). Max 5 questions per candidate.

### 5c. Squad cap / release-before-hire
- Squad cap = `RecruitmentManager.MAX_SQUAD_SIZE` (25). When full, **Hire** is blocked and the page shows a
  "release a player first" message.
- Build a small release flow: list `TeamManager.Instance.MyTeam.Players`, let the player pick one, then call
  `RecruitmentManager.Instance.HirePlayerReplacing(candidate, playerToRelease)` (releases then hires in one
  step). Or release via your existing squad screen and press Hire again.

## 7. Quick test checklist

- New game ŌåÆ Tactics: change mentality, watch stat sliders + `TacticInfoUI` move. Toggle an instruction on/off
  repeatedly ŌåÆ Familiarity dips while the new state differs from the team's habit and **recovers when you toggle
  back** (it should NOT keep dropping). Familiarity climbs over matches/training, ~3 months to ~100%.
- Add several instructions/dependencies ŌåÆ Complexity rises ŌåÆ `TacticInfoUI` shows whether the squad's **average**
  intelligence clears the bar; a smart XI shows "can handle it" even with one low-IQ player.
- Play a match ŌåÆ More Attacking/Defensive (or in-match mentality slider) ŌåÆ change takes effect next minute, line in
  the feed. At half-time ŌåÆ Edit Tactics ŌåÆ make a structural change ŌåÆ Resume Match ŌåÆ second half uses it, no restart.
- Watch for foul/booking/injury lines in the match feed; a knock barely dents morale, a long injury hits the
  squad. After a match, check a booked/injured player's `AvailabilityStatus`; confirm a suspended player is
  benched next match.
- Advance several days ŌåÆ expect the occasional off-pitch injury/illness notification + squad morale dip.
- On an interview day ŌåÆ open Recruitment ŌåÆ ask personality questions ŌåÆ watch "Personality: X / Y" narrow;
  fill the squad to 25 ŌåÆ Hire is blocked until you release someone.
- Tactics ŌåÆ toggle a dependency (e.g. "Creative Freedom") ŌåÆ picker opens with your XI; pick a player ŌåÆ toggle
  shows "Creative Freedom: Surname" and Complexity rises. Cancel the picker ŌåÆ toggle flips back off. Toggle it
  off ŌåÆ dependency removed.
- Training ŌåÆ pick a Positional drill ŌåÆ five slots appear; click an empty slot ŌåÆ picker shows the squad (minus
  any already slotted) with each player's strength/rating at the chosen position; pick ŌåÆ slot fills with name +
  X; click X ŌåÆ slot clears. "Set Training" saves the chosen players.

## 8. Player picker, reliance instructions & training slots

This wires up: a **shared player-picker popup**, **reliance instructions** on the tactics page (a reliance is now
just a property of a `TacticInstruction`), and the **five-slot** positional training selector. All written; below
is the editor build.

### Component map
| Script | Role |
|---|---|
| `TacticOptionToggle` (abstract) | shared baseline for tactic toggle buttons (Toggle + colours + event + Set/SetInteractable) |
| `TacticsToggle : TacticOptionToggle` | the instruction toggles ŌĆö now also drive reliance instructions (label shows the chosen player) |
| `PlayerPickerPopup` (+ `PlayerPickerRow`) | the shared "choose one player" modal |
| `PlayerSlotUI` | a "click to add / X to remove" player slot (training) |

> **Reliance = an instruction property.** There's no separate dependency component anymore. A `TacticInstruction`
> *becomes* a reliance the moment its **Reliance ŌåÆ Eligible Groups** list is non-empty (no separate flag ŌĆö `hasReliance`
> is derived from it); the Reliance also holds the amplified `PlayerStat`s + multiplier. It lives in
> `TacticGridLayout.tactics` like any instruction; toggling it on opens the picker to choose the reliant player.

### 8.1 The shared picker ŌĆö `PlayerPickerPopup` (REQUIRED for 8.2 and 8.3)
It's **instantiated on demand from a Resources prefab** ŌĆö no scene object to place.
1. **Row prefab** (`PlayerPickerRow`): a **Button** with a child TMP name, optionally a second TMP (per-player
   description line) and an `Image` (rating badge). Add `PlayerPickerRow` and assign **Button**, **Name Text**, and
   optionally **Sub Text** / **Rating Image**.
2. **Popup prefab** (`PlayerPickerPopup`): make its **root a Canvas** (Screen Space ŌĆō Overlay, high **Sort Order**
   so it overlays everything when instantiated standalone) containing a full-screen dim backdrop + a centred window
   (title TMP + a Scroll View). Add `PlayerPickerPopup` to the root and assign **List Container** (the Scroll View
   Content ŌĆö give it a Vertical Layout Group + Content Size Fitter), **Row Prefab**, **Title Text**, and optionally
   **Cancel Button** (a Cancel/X button, or make the backdrop a Button ŌĆö auto-wired to cancel on Awake).
3. **Put the prefab in Resources:** save it at **`Assets/Resources/UI/PlayerPickerPopup.prefab`** (exact path/name ŌĆö
   that's what `PlayerPickerPopup.Show` loads via `Resources.Load`; change the `PrefabResourcePath` const if you
   prefer another location). `Show` instantiates a fresh copy each time and **destroys it on pick/cancel** ŌĆö there's
   nothing to wire at the call sites, and no always-present scene object.

> If the prefab's root isn't a Canvas, `Show` still tries to drop it onto the first Canvas it finds ŌĆö but a
> self-contained overlay Canvas (with a high sort order) is the reliable way to guarantee it renders on top.

### 8.2 Reliance instructions on the Tactics page
A reliance is authored on a `TacticInstruction` ScriptableObject and used like any other instruction ŌĆö no new
component or panel.
1. **Author the instruction:** Create/open a `TacticInstruction` asset (`Resources/Tactics/Instructions/`).
   It becomes a reliance **simply by filling in Eligible Groups** ŌĆö there's no separate flag to tick (`hasReliance`
   is derived: any group ŌćÆ reliance, none ŌćÆ plain instruction). Under **Reliance** set:
   - **Eligible Groups** (`PlayerGroup[]`) ŌĆö which position groups may hold this reliance (union of the listed
     groups). **This is what makes it a reliance** ŌĆö leave it empty for a plain instruction. E.g. a "focus crosses
     towards" ŌåÆ `WidePlayers` + `Attackers` (no centre-backs/keepers). It does three things: **filters the picker**,
     **auto-disables** the instruction if the player later moves out of all of them, and **scopes which phases of
     play the boost applies to** ŌĆö the engine resolves position-band phases (build-up Ōēł defenders, finishing Ōēł
     attackers, transitions Ōēł the mid+def / mid+att unions), and the reliance only amplifies its stats in a phase
     whose band is *within* these groups. So a `Defenders`-only reliance lifts the build-up phase but not the
     finish; pick `MidfieldAndAttack` / `Outfield` for a reliance you want felt further up or across the team.
     (`Goalkeeper` has no match phase, so a keeper-only reliance won't bite in the sim.)
   - **Stats** ŌĆö which of the reliant player's stats count more toward the team (e.g. Passing, Creativity for a
     playmaker; Heading, Strength for a target man);
   - **Multiplier** ŌĆö extra weight on those stats (`1` = the reliant player counts double for them, `2` = triple).
   - **Familiarity Penalty** (0ŌĆō1) ŌĆö how hard a *change* of the reliant player (a sub/swap) dents familiarity.
     `0` = no effect (a minor reliance); higher = the instruction must be re-drilled.
   Add normal `statModifications` too if you want it to also shift tactic stats / Complexity.
2. **Add it to the grid:** put the asset in `TacticGridLayout.tactics`, same as any instruction (it appears as a
   toggle on whichever tab you place it). No special UI ŌĆö `TacticGridLayout` detects `hasReliance` and runs the picker.
3. Done. Toggling it on opens the picker (your **eligible** starters only); pick the reliant player and the toggle
   reads "Name: Surname". The binding is by **squad slot**, so it follows substitutions (a subbed-on player inherits
